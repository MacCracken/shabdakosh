# Security Audit — shabdakosh v3.0.0 (2026-07-05)

- **Scope**: the ported CYRIUS library `src/**` (excludes vendored `lib/`, the `rust-old/`
  oracle, and the trusted build-time generator `programs/gen_cmudict.cyr`).
- **Threat model**: shabdakosh is dictionary-first and consumed by AGNOS components (shabda,
  dhvani, vansh) that feed it **untrusted or corrupt input** — `.bin`/CMUdict/IPA/PLS/SSML/JSON
  files from disk or mmap, and arbitrary lookup words. CYRIUS is raw-memory (all-i64, `load8`/
  `store8`, no bounds checks, bump allocator), so a length/count read from untrusted input and
  used without validation is a memory-safety defect. The parity oracle (Rust) is memory-safe;
  any place the port **dropped a bound the Rust enforced** is a real bug.
- **Methodology**: a 14-agent adversarial workflow — 4 parallel finders (binary deserializer /
  text-markup parsers / UTF-8 decoder / general raw-memory bounds), each finding then
  independently **verified** (trace the path, prove a concrete reachable trigger, default to
  not-real). 10 raw findings → **9 confirmed, 1 dismissed** as unreachable.
- **Outcome**: **all 9 confirmed findings fixed and regression-tested**, plus **2 more** of the
  same class caught by the post-audit backlog review (see Addendum) — **11 fixed total**. The
  dismissed latent gap was additionally hardened. Full tree 25 suites / 677 assertions green.
  **No open findings.**

## Details & remediation

### 1–4 · Binary deserializer: unvalidated attacker-controlled lengths (HIGH)

`shabda_from_binary` / `shabda_load_binary_file` / `shabda_lazy_open` are public entry points
that decode a `.bin` dictionary from disk/mmap. The header (magic + version) was checked, but
every subsequent field — the u32 entry count, u16 word length, u16 phoneme count, and language
length — was read and used to loop/copy with **no validation against the blob length**. A blob
like `"SHBD"\x01\x00\xFF\xFF\xFF\xFF` (entry count `0xFFFFFFFF`) drove a multi-billion-iteration
loop reading far past the blob; a `wlen`/`np`/`llen` of `0xFFFF` drove a ~64 KB `load8` copy past
the end. Under `lazy.cyr`'s exactly-sized mmap this walks into the unmapped guard page →
SIGSEGV/DoS; under the `file_read_all` path it reads adjacent bump-arena heap (info disclosure /
corruption) and can still fault. The Rust oracle got this bound for free from postcard's
bounds-checked reader (`DeserializeUnexpectedEnd` → `DictParseError`); the port dropped it.

**Fix** (`format/binary.cyr`): added `_shabda_bin_oob(off, need, len)` and threaded the blob
length through `_shabda_bin_get_map` / `_shabda_bin_get_pron` and the language branch. Every
multi-byte read/copy is now guarded (`off + need <= len`); on any overrun the helpers return
`-1`, which propagates to `shabda_from_binary` → `0` (mirroring `DeserializeUnexpectedEnd`). The
per-read guards also bound the loop counts, so a bogus `n` can never over-iterate.
**Regression**: `tests/binary.tcyr` — huge u32 count, huge u16 `wlen`, and header-only truncation
all decode to `0`.

### 5, 8 · UTF-8 decoder: read/advance past the NUL terminator (HIGH / MEDIUM)

`_shabda_utf8_next` (used by `shabda_detect_script` etc.) read continuation bytes `b1/b2/b3`
without checking for the NUL terminator. A word ending in a truncated multibyte lead byte
(e.g. bytes `[0xF0, 0x00]`) read `b2`/`b3` past the NUL (OOB), and returned an advance of 2/3/4 —
walking the cursor past the terminator on subsequent calls.

**Fix** (`detect.cyr`): after each continuation-byte load, bail if it is NUL — consume only the
lead byte (return 1) and store it as an opaque code point. Because a cstring is NUL-terminated,
once a continuation byte is non-zero the next slot is guaranteed in-bounds, so all reads are now
bounded. **Regression**: `tests/detect.tcyr` — lone 2-byte and 4-byte leads decode without OOB.

### 6 · validate JSON deserializers: null Str deref (HIGH)

`shabda_validation_report_from_json` / `shabda_phonotactic_report_from_json` (and their element
codecs) did `str_cstr(json_v_str(json_v_obj_get(obj, "field")))` with no guard. A missing field
(or an unparseable root) makes `json_v_obj_get` return `0`; `json_v_str(0)` then dereferences
null. Rust's serde errored on the missing required field; the port crashed.

**Fix** (`validate.cyr`): added `_shabda_json_field_str` (guards `json_v_obj_get == 0` and
`json_v_is_str == 0`, defaulting to `""`), guarded a `0` parse root (→ return `0`), and guarded
`json_v_arr_len` on a missing array. (`format/json.cyr`'s `shabda_from_json` was already guarded
this way — hence not flagged.) **Regression**: `tests/validate.tcyr` — `"{}"`, unparseable input,
missing `language`, and a mistyped `"language":123` all decode without a crash.

### 7 · CMUdict file load: silent truncation ≥ 8 MB (MEDIUM)

`shabda_load_cmudict_file` read into a fixed 8 MB buffer; a file at/over the cap filled the buffer
and was **silently parsed as a truncated prefix** (a partial dictionary that looks valid).

**Fix**: fail loud — if the read fills the buffer (`n >= 8388607`), return `0` rather than parse a
prefix. The same guard was added to `shabda_load_binary_file` for consistency.

### 9 · parse_cmudict `@freq`: malformed value accepted (LOW)

`@freq=1e5xyz` was accepted: `f64_parse_ok` read the unbounded `input+vs` and validated only the
numeric *prefix*, ignoring trailing junk that Rust's whole-token `f32::parse()` rejects.

**Fix** (`format/mod.cyr`): parse a NUL-terminated dup of exactly the token `[vs, ve)` (so the
read can't run past the token) and reject any non-float character via
`_shabda_fmt_all_float_chars`. **Regression**: `tests/format.tcyr` — `@freq=1e5xyz` → `0`;
`@freq=0.85` still valid.

## Dismissed finding

**`cmudict.cyr:75` — region read past the pron bound.** Accurately described (the `^region`
branch read two bytes without checking `q+2 < pe`), but **not reachable under the threat model**:
`piece` is only ever the generated static string literals in `_cmudict_data_*.cyr`, produced by
the trusted `gen_cmudict.cyr`, which never emits a clipped `^` (and the current source data has
zero `@region=` annotations, so the branch is not exercised at all). The untrusted-input region
parsers use separate code (`format/mod.cyr`, `format/json.cyr`). Verdict: latent robustness gap,
not a reachable defect. **Hardened regardless** with a `q + 3 <= pe` bound (defense-in-depth for a
future data-format change), since `q` is a discarded local cursor so the guard is cost-free.

## Addendum — 2 further fixes from the post-audit backlog review

A follow-up parallel review (non-security focus) surfaced two more defects of the same
unvalidated-input class, both now fixed + regression-tested:

- **10 · HIGH — `format/json.cyr` null Str deref on a wrong-typed field.** `shabda_from_json`
  (L152) and `_shabda_pron_from_jsonv` (L106) guarded `language`/`region` with only `!= 0` +
  `json_v_is_null == 0`, then `str_cstr(json_v_str(v))`. A present-but-wrong-typed field
  (`{"language":42}`, `{"region":[]}`) makes `json_v_str` return a `0` Str → `str_cstr(0)`
  derefs null. (The audit's binary/text focus missed this JSON path.) **Fix**: gate both on
  `json_v_is_str == 1`. **Regression**: `tests/json.tcyr` — int `language` and array `region`
  decode without a crash.
- **11 · HIGH (write side) — `format/binary.cyr` fixed 2 MB serialize buffer.** `shabda_to_binary`
  `alloc(2097152)` then wrote the whole dict unchecked; a dict serializing to > 2 MB (many/long
  user entries) overflows the heap buffer. **Fix**: `_shabda_bin_total_size(dict)` computes the
  exact serialized size (mirrors the `put_*` layout) and the buffer is sized to it. This closes
  the write side to match the now-bounded read side (findings 1–4).

## Conclusion

All 9 confirmed findings are fixed and covered by regression tests; the lone dismissed item was
hardened as a precaution. The binary deserializer — the primary untrusted-input attack surface —
now bounds every field like the Rust/postcard original, and the UTF-8 and JSON paths no longer
read past their inputs. **This audit gate is met for v3.0.0.**
